Intent

這個 diff 在既有 Guided Demo 入口之上加入真正新使用者專用的四步驟 Welcome，建立一次性 eligibility、view 優先順序與安全完成流程。

Drivers

  • 空 myTrips key 不能被誤判為 returning user。
  • Welcome 必須優先於 What's New 且不能覆蓋 deep-linked Trip。
  • Demo、Create、Skip 必須有明確且不自動建立雲端資料的完成語意。
  • storage failure、重複事件、focus 與 mobile layout 需要回歸保障。

Non-goals

  • 不實作 Clone Flow
  • 不修改 FeatureTour chapters
  • 不建立 Firebase demo room
  • 不自動填入東京資料
  • 不標記 current release seen

Before / After

Before

Phase 7B-2 已有本機東京 Demo、Lobby/Settings 入口與互斥 Demo view，但沒有 first-run Welcome 或 onboarding eligibility marker。

  • 使用者可從空 Lobby 卡片或已有旅程時的 Settings 主動開啟 Demo。
  • Demo 的 Create CTA 只開啟既有空白旅程 Modal。
  • What's New 依 release seen 邏輯自行顯示，沒有 first-run session gate。

State

  • demoPreviewState 管理 Demo view
  • release state 管理 What's New 與 FeatureTour
  • 沒有 firstRunSnapshot、firstRunResolved 或 showFirstRunWelcome

Data

  • 沒有 travel-app-seen-onboarding-v1
  • Demo 仍不進入 Firebase、myTrips 或 Offline Cache
After

Phase 7B-3 在 App 初始 render 判斷真正新使用者，顯示可存取的四步 Welcome，並以一次性完成流程連到 Demo、空白 Create 或 Lobby。

  • fresh storage 與只有空 myTrips 的使用者看到 Welcome。
  • 非空 trips、release history、自訂外觀、有效 offline cache 或 member identity history 代表 returning use。
  • room deep link 先載入 Trip，返回 Lobby 後才顯示 Welcome。
  • first-run session 抑制 What's New，但不寫 release seen marker。

State

  • 新增一次性 firstRunSnapshot
  • 新增 firstRunResolved 與 showFirstRunWelcome
  • 新增 suppressReleasePromptForFirstRunSession 與 completion refs
  • Dialog 以 ephemeral step 管理四頁內容

Data

  • 完成時寫入 travel-app-seen-onboarding-v1=true
  • 讀取多組既有裝置訊號
  • Demo/Firebase/myTrips/Offline Cache 邊界維持不變
