Overview

One sentence

First-run Welcome 是只在缺乏既有使用訊號時顯示的四步驟導覽，完成後只記錄本機 onboarding marker，再依使用者選擇開啟 Demo、空白建立流程或 Lobby。

Why it exists

讓真正的新使用者先理解行程、東京唯讀範例、建立旅程與離線預覽，同時避免自動建立 Firebase 資料或與 What's New、FeatureTour 疊加。

Audience
  • 第一次使用 Travel App 的使用者
  • 維護 App 初始流程的工程師與 QA
Prerequisites
  • 理解 Lobby、Demo Preview 與真實 Trip 是互斥 App views
  • 知道 localStorage 是裝置狀態而非授權

User journey

  1. 1

    判定是否為真正新使用者

    App 在寫入空的 myTrips 預設值前建立一次 eligibility snapshot；任一既有使用訊號都會阻止自動 Welcome。

  2. 2

    顯示四步驟 Welcome

    符合資格且沒有 room deep link 或其他互斥 view 時顯示 dialog，What's New 與 FeatureTour 不同時顯示。

  3. 3

    閱讀產品概念

    四步依序介紹集中管理、東京唯讀範例、建立自己的旅程、多人協作與唯讀離線預覽。

  4. 4

    選擇完成動作

    使用者可開啟東京 Demo、開啟既有空白建立 Modal，或略過並留在 Lobby；三者都先完成 onboarding。

  5. 5

    後續 session 行為

    當次 first-run session 不立刻顯示 What's New；重新載入後 Welcome 不再顯示，未讀的 release flow 可恢復。

State flow

firstRunSnapshot

Owner: TravelApp

Initial: App 首次 render 時由 readFirstRunEligibilitySnapshot 產生，之後不重新讀取。

  • App mount: 未建立 → 固定 snapshot. 避免 App 自己寫入空 myTrips 後改變資格。

firstRunResolved

Owner: TravelApp

Initial: 非 first-run session 為 true；first-run session 為 false。

  • 任一完成動作: false → true. 本 session 不重新開啟 Welcome，即使 storage write 失敗。

showFirstRunWelcome

Owner: TravelApp

Initial: 符合 first-run 且沒有 room deep link 時為 true。

  • 延後條件解除: false → true. 返回 Lobby 後顯示 Welcome 並關閉 What's New。
  • 完成 onboarding: true → false. 先卸載 dialog，再進入下一個 view 或 modal。

Dialog step

Owner: FirstRunWelcomeDialog

Initial: 0；不合法 initialStep 也回到 0。

  • Next: 0-2 → 1-3. 只切換本機 ephemeral step，不完成 onboarding。
  • Back: 1-3 → 0-2. 只回上一頁。

Data flow and side effects

read

localStorage existing-use keys → firstRunSnapshot

Persistence: 既有裝置資料，不由 Welcome 建立

Side effect: 無；讀取失敗時使用安全 fresh fallback。

read

window.location.search room → firstRunSnapshot.hasRoomDeepLink

Persistence: URL view state

Side effect: 只延後顯示，不改 eligibility。

Side effect: 下次載入不再顯示 Welcome。

Side effect: 切換到唯讀 Demo；不建立 room。

Side effect: 顯示既有空白建立 Modal；點 CTA 本身不寫 Firebase。

Read / Write boundaries

Reads

  • travel-app-seen-onboarding-v1
  • google-travel-my-trips
  • travel-app-seen-release-*
  • google-travel-custom-bg
  • google-travel-offline-trip-cache-v1
  • travel-active-member-* and travel-checklist-actor-*
  • URL room query

Writes

  • travel-app-seen-onboarding-v1 = true after Demo, Create, Skip, or Escape

Must not write

  • Firebase Realtime Database rooms
  • Firebase Storage
  • google-travel-my-trips Demo entries
  • Offline Trip Cache
  • current release seen marker
  • production data or rules

External systems

  • Browser localStorage
  • Browser sessionStorage for release flow
  • Browser URL query

Invariants

  • 空的 google-travel-my-trips 陣列本身不代表 returning user。
  • room deep link 只延後 Welcome，不能把 onboarding 永久標記為完成。
  • Next 與 Back 不寫 onboarding marker。
  • Demo、Create、Skip 的完成 callback 在 App 與 Dialog 層都只執行一次。
  • First-run session 不標記 current release seen，也不與 What's New 或 FeatureTour 疊加。
  • storage write 失敗時，本 session 仍 resolved 且使用者選擇繼續執行。

What it does not do

  • 不自動建立 Firebase room 或複製東京範例。
  • 不把 Demo 放入 myTrips 或 Offline Trip Cache。
  • 不提供完整離線編輯或背景同步。
  • 不把裝置 member identity 當成授權。
  • 不修改 FeatureTour chapters 或自動標記 release seen。

Architecture decisions

ADR-001 · accepted

Built-in Demo 使用本機資料，不寫 Firebase Database 或 Storage。

ADR-002 · accepted

Built-in Demo 不加入代表真實旅程參照的 myTrips。

ADR-004 · accepted

Offline Preview 是唯讀快照，不支援離線 mutation replay。

ADR-005 · accepted

Demo 與 first-run phases 不修改既有 FeatureTour chapters。

Risks

high

localStorage 訊號只能判斷裝置使用歷史，不能作為身分或授權證據。

Mitigation: 保持 onboarding eligibility 與 Firebase authorization 完全分離。

medium

storage 完全不可用時安全 fallback 會把裝置視為 fresh，可能再次顯示 Welcome。

Mitigation: App 以 firstRunResolved 在同一 session 防止重開，並將失敗視為非致命。

medium

App.jsx 同時協調 release、Demo、Offline、Trip 與 modal，優先順序變更容易造成疊加回歸。

Mitigation: 維持 integration 與 E2E 排他性測試。

low

真實行動裝置的虛擬鍵盤與 safe-area 行為仍可能不同於 Playwright Mobile Safari。

Mitigation: 保留實機 QA，不以模擬專案取代。

Known unknowns

Impact: 可能影響小螢幕上完成 CTA 的可達性。

Next step: 維持目前非致命行為，若需改善須另做隱私與儲存策略決策。

Glossary

Eligibility snapshot
App 初始 render 只建立一次的裝置使用訊號集合，避免 mount effects 改變判定。
First-run resolved
本 session 已完成或不需要 first-run 的 React state；它不等同於 storage marker 寫入成功。
Deep-link deferral
先允許 room view 載入，等返回 Lobby 才顯示尚未完成的 Welcome。
Built-in Demo
不建立 Firebase room、不加入 myTrips、不寫 Offline Cache 的本機唯讀東京範例。
What's New
以 release seen key 管理的版本更新說明；first-run session 只抑制，不標記已讀。

Quiz

1. 哪個條件本身不會取消 first-run eligibility？

Evidence: E4, E6

hasRoomDeepLink 只控制初始顯示時機，shouldShowFirstRunOnboarding 不把它列為 returning signal。

2. 使用者選擇查看東京 Demo 時，哪項資料會被持久化？

Evidence: E9, E15, E18

完成流程只嘗試寫 onboarding marker；Demo view model 保存在 React state。

3. 為何 eligibility snapshot 必須在 App 初始 render 建立一次？

Evidence: E7, E14

App mount 後會保存空 myTrips；一次性 snapshot 保留 mount 前的真實使用訊號。

4. Skip 在同一 session 對 What's New 的影響是什麼？

Evidence: E8, E15

first-run session gate 持續抑制 release prompt，integration test 也確認 mark seen 未被呼叫。

5. onboarding marker 寫入失敗時，App 應如何處理？
